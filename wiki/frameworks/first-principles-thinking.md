# First Principles Thinking

## Overview
A decision-making approach that requires deriving solutions from fundamental building blocks rather than copying or adapting existing patterns. The core practice is identifying path-dependence in existing solutions — which assumptions were true when those solutions were designed that may no longer be true today — and then re-deriving optimal answers from scratch given current constraints.

## Components / Steps

### 1. Identify Path-Dependence
Every existing solution carries accumulated constraints from decisions made at its founding. Ask: what assumptions were true when this was designed that may no longer hold? Look for changes in:
- Technology capabilities (e.g., LLMs making previous NLP approaches obsolete)
- Cost structures (e.g., cloud computing changing infrastructure economics)
- Context (e.g., COVID changing whether "people can leave the house" as a Boolean)
- Market conditions

### 2. Derive From Available Building Blocks
List the actual building blocks available today — technologies, market conditions, team capabilities, regulatory environment. Design the optimal solution for *this* set of building blocks, not the set that existed when the incumbent solution was created.

### 3. Check Against the Status Quo
After independent derivation, compare with the existing solution. Two outcomes:
- If you arrive at the same answer: confirms the existing approach has enduring validity
- If you arrive at a different answer: you've found a genuine opportunity for improvement

### 4. Apply the Thermostat Test (Tobi's framing)
Re-run the optimization function over the current state. Would you make the same product/process/organizational decision if you were designing it from scratch today? If not, the gap represents debt to be addressed.

## When to Use
- Evaluating whether an existing product category is ripe for disruption
- Making organizational or structural decisions (e.g., remote vs. in-person)
- Designing new products or features where existing patterns exist but may be suboptimal
- When receiving pitches that are "a good version of X" — apply first principles to test whether the category even needs the product

## Tobi Lütke's Application
Tobi applies this continuously at Shopify:
- **Shopify's founding**: E-commerce software in 2004 was path-dependent on serving existing retailers with complex business logic. Re-deriving from scratch for "frustrated people in dead-end careers who can spare a lunch break" produced a radically simpler product that ultimately outcompeted enterprise players on all dimensions.
- **Remote work decision**: COVID changed one foundational Boolean (can people leave the house?). Re-running the optimization function from that changed assumption produced a completely different optimal answer — fully remote — rather than an incremental adjustment.
- **Monorepo decision**: A consequential architectural choice (monorepo vs. multi-repo) that is wrong at certain company sizes and right at others. Being personally in the code enables Tobi to hold genuine conviction on this.
- **Avoiding "do a good version of X"**: If you want to do something better than what exists, you must do it differently. Anything else fails Archimedean logic.

## Relationship to Other Concepts
- **Goodhart's Law / Overfitting**: Both are instances of the same universal truth — optimizing for a proxy destroys the proxy's validity. First principles thinking identifies when you're optimizing for a proxy.
- **Positional vs. tactical game**: Tactics are derived from position; first principles thinking is how you build position.
