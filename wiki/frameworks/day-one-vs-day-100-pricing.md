# Day One vs. Day 100 Pricing

> A framework for deciding which features belong in your free or entry tier vs. your premium tier, based on when in a user's lifecycle they actually get value from the feature.

## Overview

The most common mistake in SaaS pricing is creating a premium tier that includes features users don't understand or need when they first sign up — and a free tier that gives away features advanced users would happily pay for. The Day One vs. Day 100 framework cuts through this by asking a simple question for every feature: when does a user actually derive value from this?

Naomi Ionita developed this framing at Invoice2go, where applying it led to a rare compound win: doubling the upgrade rate from starter to pro plan while simultaneously increasing the pro plan price by ~30%. Getting twice as many people to upgrade while paying more is extremely unusual in pricing optimization.

## Components / Steps

**Day One Features:** Premium capabilities that deliver immediate, tangible value to a brand-new user in their very first session. These are features that justify being put behind a paywall from the beginning — users can see and feel the value right away.

Examples of Day One features:
- A core analytical capability (e.g., advanced reporting that a new user can run immediately)
- A collaboration feature that works from the first moment two people use the product together
- A workflow automation that saves time from the very first use

**Day 100 Features:** Advanced functionality whose value is only realized after the user has significant data, team adoption, or organizational scale built up in the platform. Putting these in your entry tier wastes cognitive load — users don't understand why they're paying for something they can't use yet.

Examples of Day 100 features:
- Enterprise SSO and security features (value realized when IT gets involved at scale)
- Advanced analytics requiring months of historical data to be meaningful
- Bulk management tools that only matter once you have hundreds of records
- Admin controls for large teams (meaningless to a solo user)

## How to Apply

1. **List all your features.** Include both existing and planned.

2. **Sort each feature into Day One or Day 100.** Ask: "If a user just signed up today, can they immediately see the value of this feature?" If yes: Day One. If they need to use the product for weeks or months first: Day 100.

3. **Structure your pricing tiers accordingly.** Day One features can go in a premium entry tier (or behind the initial paywall in freemium). Day 100 features belong in a higher tier that users upgrade to over time as their usage matures.

4. **Design natural upgrade triggers.** When a user's usage or team size reaches the threshold where Day 100 features become valuable, create in-product nudges to the next tier. The upgrade path should feel natural — not a surprise.

5. **Keep the pricing page simple.** The biggest mistake on pricing pages is listing everything. Users can't evaluate 40 feature differences. Anchor the tier description on the one or two features that are the actual conversion drivers (identify these via the 100-point survey or Van Westendorp research).
